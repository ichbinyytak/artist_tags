# Database Build Rules Draft

记录时间：2026-05-23

这份文件是把原有数据库规则，与最近关于正式索引表、重复清理、标签表体积控制、歌曲贴标用途等讨论结果合并后的草案版。

它的目标不是直接替换原文件，而是先给出一份更接近当前实际状态的统一说明，方便后续决定要不要覆盖正式规则文件。

---

## 一、当前最重要的结论

当前数据库体系里，最核心、最需要保证正确的正式结果，不是平台原始库，也不是过程文件，而是正式跨平台索引表：

- `backups/index-table/formal-index-artist-keys.json`

原因：

- 原始平台数据库后续可以重新下载、重新抓取
- 过程文件可以删除或重跑
- 真正难以替代、最消耗人工时间的是跨平台匹配关系本身

因此，当前正式数据库建设的第一优先级是：

- 保证正式索引表尽量准确
- 保证正式索引表尽量轻
- 让后续所有正式业务表都围绕这张表展开

---

## 二、当前目标

- 数据库优先追求体积小、搜索快、更新快、抓取快
- 正式业务表尽量轻量，避免把低频信息和高频信息混在一起
- 正式主线以可维护、可重建、可验证为核心
- 后续主要用途不是“给人搜歌手”，而是“通过歌曲文本中的歌手名，给歌曲打上歌手标签”

---

## 三、总体原则

- 高频更新和低频更新分开
- 能不重复存的信息就不重复存
- `artist_key` 作为统一主关联键
- 平台原始数据尽量保留在源表，正式业务表尽量瘦身
- 搜索/命中能力和主业务表分离
- 月更流程只关注真正会变化的数据
- 标签结果尽量编码化，减少长文本重复存储

---

## 四、当前正式主线

当前正式主线可以理解为以下几层：

1. 正式索引表
2. 轻标签表
3. 名字命中索引表
4. QQ 月更表
5. 网易月更表
6. 冷数据表
7. 单平台剩余库

其中：

- 正式索引表是主干
- 标签表服务于歌曲贴标
- 名字命中索引表服务于从歌曲文本识别歌手
- 月更表服务于高频统计更新
- 冷数据表只保留低频补充信息

---

## 五、`artist_key`

### 当前规则

- `artist_key` 由 QQ ID 和网易 ID 组成
- 两个平台都存在时，`artist_key` 同时包含两个平台 ID
- 只有单平台时，只保留该平台 ID
- `artist_key` 的目标是唯一、可读、方便人工核对

### 当前正式要求

- 正式索引表中的双平台合并结果，最终应尽量收敛到 1 对 1
- 同一个 QQ 不应绑定多个网易
- 同一个网易不应绑定多个 QQ
- 如果后续发现错配，优先修正式索引表，而不是在下游表里打补丁

---

## 六、正式索引表

正式冻结结果：

- `backups/index-table/formal-index-artist-keys.json`

### 定位

- 全库最核心的正式跨平台对应关系表
- 后续标签表、月更表、名字命中表都以它为主关联层

### 当前原则

- 尽量轻量
- 只保留正式身份关系
- 不承担标签、统计、搜索的完整职责

### 建议字段

- `artist_key`
- 必要时保留少量来源批次信息，例如 `merge_batch` / `merge_batches`

### 不建议放入

- 大量别名
- 原始标签
- 月更统计
- 复杂来源说明
- 低频网页快照

---

## 七、轻标签表

### 定位

- 正式静态标签主表
- 主要用于后续歌曲贴标

### 当前用途理解

这张表的核心作用不是展示平台原始详情，而是：

- 给每个 `artist_key` 提供稳定、精简、可复用的歌手标签结果

### 当前原则

- 标签表不重复保存平台 ID
- 标签表不保存主姓名或别名，名字查找统一由名字命中索引表承担
- 标签表以静态字段为主
- 不混入高频统计
- 不混入大段来源说明

### 建议字段

- `artist_key`
- `gender_code`
- `region_code`
- `language_codes`
- `style_codes`

### 地区与语言的分工

这里建议把“地区”和“语言”明确拆开，不混成一个字段。

#### `region_code`

表示歌手主要归属地区或市场来源，例如：

- 中国
- 日本
- 韩国
- 欧美
- 其他地区

这里的“其他地区”很重要，主要用于表示：

- 新加坡
- 马来西亚
- 其他不适合直接归到中国 / 日本 / 韩国 / 欧美，但实际会发中文歌的歌手

也就是说：

- 地区字段负责“这个歌手主要来自哪里”
- 不直接代表“他唱什么语言”

#### `language_codes`

表示歌手主要使用的语言，可以多选，例如：

- 中文
- 粤语
- 闽南语
- 日语
- 韩语
- 英语

如果一个歌手长期跨语种发歌，可以同时保留多个语言代码。

### 当前建议

- 地区和语言必须分开存
- 中文不等于中国地区
- 粤语、闽南语不应简单并到“中文”一个值里
- 新加坡、马来西亚等歌手如果主要发中文歌，建议：
  - `region_code = 其他地区`
  - `language_codes` 里仍可写 `中文` / `粤语` / `闽南语`

### 不建议直接放入

- 主姓名和别名
- 月更统计
- 平台原始 ID
- 原始标签全文
- 大量归一化搜索字段

### 标签来源与外部补全规则

- 所有进入正式索引的艺人都进行外部多来源检索，即使 QQ 或网易已经提供原始标签也不跳过；标签表为低频生成表，完整性优先于单次生成速度。
- QQ、网易等平台原始自带标签是重要证据输入，应回收并统一映射，但不再作为停止外部搜索的条件。
- 中国艺人重点检索百度百科、中文 Wikipedia 与可用的结构化音乐来源；日本、韩国和欧美艺人重点检索当地语言 Wikipedia 与结构化音乐来源。
- 可访问且身份可核验的结构化来源，例如 MusicBrainz、Apple Music，以及适合相应地区的补充来源，可以与平台原始标签并行收集；同名艺人必须先完成身份匹配。
- 同一艺人可以合并多个被不同来源支持的标准风格代码，而不是只保留单一主风格；平台叫法不同但含义相同的标签统一落到同一个标准编号。
- 外部检索和平台原始结果都只把最终统一代码写入轻标签表；来源 URL、原始词、证据文本、抓取日期、匹配状态保存在冷数据或证据表。
- 艺人介绍直接写明风格、类型或乐队属性时，可作为标签候选依据，例如“民谣男歌手”“摇滚乐队”。
- 艺人介绍将其代表作、出道作或持续性作品明确描述为某种风格时，也可作为艺人标签候选依据，例如以“嘻哈说唱单曲”作为出道定位的艺人。
- 单次合作、单场演出、致敬作品、仅说明某首歌曲融合元素的描述，不单独作为艺人全局标签依据。
- 无法明确落到标准风格代码的来源词进入待复核/待扩充字典状态，不应勉强合并到过宽的标签。
- 日本艺人标签生成时，日文平台风格词应先进入统一风格别名表再转换代码；稳定的动漫/动画音乐标签统一使用 `12.04`，低价值长尾无风格行可按当前周期延期，不追求 100% 覆盖。
- 韩国艺人标签生成时，韩文平台风格词应先进入统一风格别名表再转换代码；稳定的 Trot 标签统一使用 `11.18`，韩国传统音乐使用 `11.19`，低价值长尾无风格行可按当前周期延期。

### 中文歌手标签表当前收尾策略

中文歌手轻标签表当前不追求 100% 风格覆盖。全量外部检索和规则补全后，正式标签表只保留有 `style_codes` 的艺人：

- 有热度、歌曲量或其他本地信号的高价值剩余艺人，单独进入最后一轮复查。
- 低价值长尾艺人如果缺少热度、歌曲量、QQ raw tag、平台原始标签和可靠外部证据，不再保留在正式标签表中。
- 后续如果出现新艺人或歌曲侧命中特定艺人，再单独搜索、匹配并补入正式标签表。

## 八、标签编码策略

为了控制 7 万级规模下标签表的体积，当前建议：

- 地区、性别、语言、风格尽量编码化
- 文本说明放在单独字典表

### 例子

- 地区：`1 = 中国`，`2 = 日本`，`3 = 韩国`，`4 = 欧美`
- 性别：`m = 男`，`f = 女`，`g = 组合`，`u = 未知`
- 语言：例如 `10 = 中文`，`11 = 粤语`，`12 = 闽南语`，`13 = 日语`，`14 = 韩语`，`15 = 英语`
- 风格：使用 `data/platform-source/qq/v8/dictionaries/standard-style-taxonomy.json` 中的分层字符串编号，例如 `01.02 = 华语流行`、`05.01 = 嘻哈/说唱`

这里的地区编码也建议预留“其他地区”，例如：

- `1 = 中国`
- `2 = 日本`
- `3 = 韩国`
- `4 = 欧美`
- `5 = 其他地区`

### 推荐实现

标签结果表只放代码：

- `artist_key`
- `gender_code`
- `region_code`
- `language_codes`
- `style_codes`

字典表单独存：

- `region_code -> 中文说明`
- `gender_code -> 中文说明`
- `language_code -> 中文说明`
- `standard_style_id -> 中英文说明`

### 这样做的原因

- 减少长文本重复
- 降低标签表体积
- 便于后续统一改名
- 便于歌曲侧直接继承编码结果

## 九、名字命中索引表

原规则里偏向叫“姓名搜索索引表”，但按当前用途，建议理解为：

- 歌曲贴标姓名索引表
- 歌手名字命中表

### 当前用途

未来主要不是给人搜歌手，而是：

- 从歌曲名、文件名、描述文本中识别歌手名
- 再映射到 `artist_key`
- 再给歌曲继承歌手标签

### 定位

- 这是一张名字到 `artist_key` 的命中映射表
- 它不负责正式身份定义
- 它只负责“文本里出现这个名字时，程序大概率该命中谁”

### 建议字段

- `name_key`
- `raw_name`
- `artist_key`
- `source`
- `priority`
- `is_primary`
- `lang`

### 说明

- `name_key` 是归一化后的名字键
- `raw_name` 是原始可读名字
- `source` 说明来自 QQ 主名、网易主名、别名等
- `priority` 用于命中排序

### 原则

- 这张表可以大
- 但结构必须简单
- 不要让标签主表承担这部分职责

## 十、QQ 月更表 / 网易月更表

### 当前原则

- 月更表不重复保存平台 ID 和静态字段
- 月更表只保留最新结果，不保留历史月份
- 月更表只保留高频变化字段

### 建议字段

- `artist_key`
- `album_count`
- `song_count`
- `fan_count`
- `updated_at`

### 当前正式口径

- QQ 正式月更表覆盖 QQ v8 分区源表中的艺人
- 网易正式月更表只覆盖 `data/platform-source/netease/netease-artist-stats-backup.json` 中的艺人
- 不要用 `data/formal-index/formal-index-artist-keys.json` 生成网易正式月更表；formal index 会引入 QQ 绑定行和重复网易 ID
- 网易 formal-index 全覆盖结果只能作为参考表保留，不作为正式月更主线

### 必备依赖

- QQ 月更依赖 `artist_mid`
- 网易月更依赖 `data/platform-source/netease/netease-artist-stats-backup.json` 的最新抓取结果

### 网易月更固定命令

```bash
node scripts/netease/build-netease-monthly-from-stats-backup.mjs \
  --input data/platform-source/netease/netease-artist-stats-backup.json \
  --output data/monthly/netease/netease-monthly.json
```

## 十一、冷数据表

### 定位

- 保存低频静态详情
- 保存原始标签
- 保存不适合放进轻标签表的补充信息

### 建议字段

- `artist_key`
- `qq_raw_tags`
- `netease_raw_tags`
- `detail_snapshot`
- `updated_at`

### 原则

- 可以重
- 但不要进入正式主业务高频链路

## 十二、单平台剩余库策略

### 当前结论

- 剩余未匹配艺人可以保留
- 但不应继续默认进入正式主线

### 当前规则

- 网易剩余未匹配艺人单独作为剩余库保留
- QQ 或其他单平台长尾艺人如果未正式进入索引体系，也可按同样思路保留
- 剩余库默认不加入正式月更表
- 只有后续业务明确命中时，才按需补合并

### 理解

- 剩余库是“保留层”
- 不是“正式主表层”

---

## 十三、关于 7 万规模的判断

### 当前判断

- 正式索引表到 7 万左右是可以接受的
- 轻标签表到 7 万左右也是可以接受的

### 风险不在记录数本身，而在字段膨胀

真正危险的是：

- 别名过多
- 标签文本反复重复
- 主表兼做搜索表
- 把低频原始信息混进正式轻表

### 因此当前建议

- 不按地区拆标签表
- 而是按用途拆层
- 主标签表尽量轻
- 搜索/命中索引单独拆出
- 标签尽量编码化

---

## 十四、当前推荐的正式路径

### 1. 正式索引表

- 只保留跨平台对应关系

### 2. 轻标签表

- 只保留静态标签结果
- 标签优先编码化

### 3. 名字命中索引表

- 专门服务于歌曲贴标时的歌手名识别

### 4. 月更表

- 单独保留动态统计

### 5. 冷数据表

- 放低频原始和补充信息
