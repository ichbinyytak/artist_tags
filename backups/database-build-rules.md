# Database Build Rules

记录时间：2026-05-21

这份文件不是最终锁死的技术规范，而是当前数据库建设阶段的方向说明。它服务于后续建库决策，目的是把已经讨论出的原则、分层思路和当前决定沉淀下来，避免每次都从口头重新整理。

后续如果发现更优方案，应直接在这份文件基础上迭代，而不是把它视为不可修改的定稿。

## 目标

- 数据库优先追求体积小、搜索快、更新快、抓取快。
- 当前阶段目标是先把数据库结构规划清楚，不直接进入歌单逻辑设计。
- 正式业务表尽量轻量，避免把低频信息和高频信息混在一起。

## 总体原则

- 高频更新和低频更新分开。
- 能不重复存的信息就不重复存。
- `artist_key` 作为统一主关联键。
- 平台原始数据尽量保留在源表，正式业务表尽量瘦身。
- 搜索能力和业务表分离，不把搜索需求直接堆到主业务表里。
- 月更流程只关注真正会变化的数据。

## 当前确定的表层设计

当前数据库按以下几层理解：

1. 标签 ID 索引表
2. 标签表
3. QQ 月更表
4. 网易月更表
5. 冷数据表
6. 姓名搜索索引表

这些表目前只是建设方向，不代表已经全部实现。

## 当前已定规则

### `artist_key`

- `artist_key` 由 QQ ID 和网易 ID 组成。
- 两个平台都存在时，`artist_key` 同时包含两个平台 ID。
- 只有单平台时，只保留该平台 ID。
- `artist_key` 的设计目标是唯一、可读、方便人工核对。

### 标签 ID 索引表

- 索引表只保留最轻量的信息。
- 索引表姓名只留一个，QQ 优先，网易兜底。
- 索引表主要服务于全库关联和快速定位，不承担完整标签和搜索职责。

### 标签表

- 标签表不重复保存平台 ID。
- 标签表保留双平台姓名，便于核对和后续维护。
- 标签表以静态信息为主，例如性别、地区、语言、风格。
- 风格标签目前不区分来源，先强调统一和精简。

### 月更表

- 月更表不重复保存平台 ID 和静态字段。
- 月更表只保留最新结果，不保留历史月份。
- 月更表只保留高频变化字段，例如专辑数、歌曲数、粉丝数和更新时间。

### QQ 标签策略

- QQ 标签只首次抓取。
- 后续 QQ 月更流程不再抓标签。
- 低频静态标签信息应与月更结果分开保存。

### 搜索策略

- 搜索能力和业务表分离。
- 主业务表不为了搜索而塞入大量别名字段。
- 搜索相关的数据应单独整理到姓名搜索索引层。

## 暂未定事项

以下内容后续还要继续讨论和调整：

- 标签抓取与映射的最终方案
- 冷数据表的保留粒度
- 姓名搜索归一化规则细节
- 后续是否补统一月更总表
- 未来是否增加歌曲层标签库

- 这份文件是当前方向说明。
- 它服务的是后续建库决策，不是最终定稿规范。
- 它的价值在于减少重复讨论，并为后续实现提供统一参照。

## 使用方式

- 每次准备新建表、改字段、补抓取逻辑之前，先回看这份规则文件。
- 如果某次讨论明确推翻了旧规则，应直接更新本文件，而不是只停留在聊天记录里。
- 如果后面某张正式表的实现与本文件不一致，应优先检查是规则需要更新，还是实现偏离了方向。
